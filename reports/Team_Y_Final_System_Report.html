1. Introduction

College0 is a full-stack academic operations system: visitors explore public statistics and apply for admission; registrars advance semester phases, define classes, resolve applications and complaints; students enroll under capacity and conflict rules, write reviews, use AI assistance scoped to their enrollments; instructors manage waitlists, grade, and file complaints. The implementation centers on Supabase (PostgreSQL with row-level security, Auth, Edge Functions, pgvector) plus a React 19 single-page app built with TypeScript, Vite, Tailwind CSS v4, and React Router v7. Optional hosting uses Vercel for the static frontend; database and server logic remain on Supabase.

Install & run from scratch
  1. Clone the repository (see §4). Open college0-app/.
  2. npm install then copy .env.example → .env and set VITE_SUPABASE_URL, VITE_SUPABASE_ANON_KEY.
  3. Create a Supabase project; enable uuid-ossp and vector. Apply SQL migrations in college0-app/supabase/migrations/ in filename order (see root README.md).
  4. Add SUPABASE_SERVICE_ROLE_KEY to .env; run npm run seed:registrar for the first registrar account (or create a user in the Dashboard with metadata role: registrar).
  5. Deploy Edge Functions from college0-app/ with the Supabase CLI; set secrets GEMINI_API_KEY, etc., per README.
  6. npm run dev — SPA at localhost. Production: connect the GitHub repo to Vercel with root college0-app.

5. Database schema (core)

PostgreSQL schema public is defined incrementally by ordered migrations. Auth identities live in auth.users; public.profiles mirrors each user with role, academic fields, and status. Below is a logical ER sketch of principal entities and foreign keys.

auth.users (id) │ 1:1 ▼ profiles (id PK/FK, role, full_name, student_id, status, cumulative_gpa, …) │ ├──< semesters (id) ──< classes (id, semester_id, instructor_id→profiles, │ course_code, title, schedule_time, max_students, │ avg_rating, is_cancelled, location_lat/lng/label, …) │ │ │ └──< enrollments (id, student_id→profiles, class_id→classes, │ status, grade, semester_id, …) │ ├──< applications (visitor/instructor pipeline) ├──< reviews (class_id, author_id, stars, body, filtered_body, is_hidden) ├──< warnings (target_id, reason, semester_id, …) ├──< complaints (filed_by, against, …) ├──< graduation_applications ├──< study_groups ├──< taboo_words ├──< fines ├──< document_embeddings (content, embedding vector(768), …) └──< audit_log, required_courses, student_semester_honors, …

Security model

Row Level Security (RLS) policies on tables such as profiles, classes, enrollments, reviews, and complaints enforce role-based access; sensitive operations (phase transitions, scans, grade posting, etc.) are additionally guarded by SECURITY DEFINER RPCs checking _current_role().
